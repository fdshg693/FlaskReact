# インフラ・構造に関する課題

プロジェクト構造、データ配置、設定ファイルに関する課題を管理します。

---

## INF-001: data フォルダの散在

| 項目 | 内容 |
|------|------|
| **ID** | INF-001 |
| **優先度** | 中 |
| **ステータス** | 未着手 |

### 現状
`data`、`output`、`tmp` の3つのフォルダが分散して存在している。

### 改善案
以下のように統一したフォルダ構成にまとめる：
- `data/source` - 入力データ
- `data/output` - 出力データ
- `data/tmp` - 一時ファイル

#NOTE: フォルダ構成の統一が必要
理由: 現状ではデータの配置場所が分散しており、新規参入者やAIツールがプロジェクト構造を把握しにくい

---

## INF-002: 設定ファイルの散在

| 項目 | 内容 |
|------|------|
| **ID** | INF-002 |
| **優先度** | 中 |
| **ステータス** | 未着手 |

### 現状
設定ファイルがフォルダ・サービスごとに分かれており、形式や配置場所が統一されていない。

### 改善案
- 設定ファイルの命名規則を統一
- 配置場所のルールを明確化
- 設定ファイルの形式（YAML/JSON/TOML）を統一

#NOTE: 設定ファイルの管理方針を明確化すべき
理由: 形式や配置場所がバラバラだと、設定の更新漏れや誤った設定の適用リスクが高まる

---

## INF-003: 開発規約の整備

| 項目 | 内容 |
|------|------|
| **ID** | INF-003 |
| **優先度** | 中 |
| **ステータス** | 未着手 |

### 現状
開発規約が十分に整備されていない。

### 改善案
- コーディング規約の整備
- コミット・PR ルールの明確化
- レビュープロセスの標準化

#NOTE: docs/dev_contract の内容を拡充すべき
理由: 開発規約が曖昧だと、コードの一貫性が損なわれ、レビュー時の判断基準も不明確になる
